Fundamentos de Ingeniería de Software

Live Coding Edition

Domina los Principios S.O.L.I.D.

Bienvenidos al lado oscuro del código espagueti. Hoy vamos a usar IA para crear un monstruo monolítico y luego, paso a paso, lo vamos a curar usando principios de diseño, TDD y DDD. ¿Listos para refactorizar?

Fase 1: La Trampa (Creando el Monstruo)

Abre tu IDE favorito, crea un nuevo proyecto en Java (trabajo individual) y utiliza la herramienta Antigravity con el siguiente prompt exacto.

"Calculadora SoLiD. Quiero crear un programa en java que tenga operaciones aritmeticas entre numeros enteros, de tipo binarias (suma, resta, etc) y unarias (raiz cuadrada, logaritmo natural). Es una aplicación de consola. Tip: quiero todo en la misma clase"

Acción Requerida:

Sube esta clase inicial a tu repositorio personal en GitHub/GitLab, haz tu primer commit (feat: codigo inicial espagueti) y envía el link en la plataforma del curso antes de continuar.

Fase 2: Reflexión Crítica

Mira el código que acaba de generar la IA. Responde las siguientes preguntas en los comentarios de tu commit o en tu archivo README.md:

01.

¿Qué responsabilidades tiene exactamente esta clase?

02.

Si cambio la forma de imprimir en consola, ¿qué pasa con las operaciones?

03.

Si quiero tener 2 menús distintos, ¿qué debería modificar?

04.

¿Cómo adiciono operaciones nuevas (ej. Potencia)? ¿Rompe algo más?

05.

¿Qué pasa si ingreso validaciones de dominio (ej. no dividir por cero, no logaritmos negativos)? ¿Dónde irían en este desastre?

Fase 3: Live Refactoring (La senda SOLID)

A medida que el profesor explique cada letra del principio S.O.L.I.D. en clase, tu misión será refactorizar tu código utilizando prompts iterativos.

  • Regla de oro: Antes de aplicar un nuevo principio, crea un nuevo paquete en tu proyecto (ej. solid.s, solid.o, etc.) y mueve/copia lo nuevo allí.
  • TDD al rescate: Pídele a tu IA que primero genere las pruebas unitarias para la funcionalidad refactorizada antes de implementar la clase.
  • Piensa en DDD: Extrae reglas de negocio en Value Objects (ej. validar que un número sea positivo para una raíz cuadrada).
S - Single Responsibility O - Open/Closed L - Liskov Substitution I - Interface Segregation D - Dependency Inversion

Entregables

  • Link a tu repositorio de código (Asegúrate de que sea accesible).
  • Un paquete con la clase inicial monolítica.
  • Un paquete con la versión final refactorizada aplicando todos los principios S.O.L.I.D.
  • Evidencia de uso de TDD (clases de test unitarios presentes).
  • Evidencia de DDD (separación de capa de dominio y validaciones).
  • Material adicional: Un documento (Markdown o PDF) con el resumen de tu interacción con Antigravity (prompts utilizados, errores que corrigió la IA, y cómo la guiaste paso a paso).
  • Nota: La calculadora corre en consola y no requiere persistencia de datos (bases de datos).

Rúbrica de Evaluación

Criterio Novato (0.0 - 2.5) Competente (2.6 - 4.0) Maestro Jedi (4.1 - 5.0)
Reflexión Inicial No responde las preguntas o las respuestas son muy superficiales. Responde la mayoría de las preguntas pero le falta profundidad en el análisis de responsabilidades. Análisis crítico profundo. Identifica claramente los antipatrones y el impacto del acoplamiento.
Aplicación S.O.L.I.D. El código final sigue estando altamente acoplado. Omite más de 3 principios. Aplica correctamente SRP y OCP, pero presenta fallas leves en Liskov, ISP o inyección de dependencias. Arquitectura impecable. Uso de interfaces correcto, inyección de dependencias y código fácilmente extensible.
TDD y Testing No hay pruebas unitarias o no compilan. Existen pruebas básicas de caminos felices, pero no fueron construidas iterativamente ni cubren errores. Excelente cobertura. Evidencia de diseño guiado por pruebas. Cubre casos borde (ej. división por cero).
Principios DDD Toda la lógica de validación está en la consola o métodos genéricos. Intenta separar el dominio pero hay fugas de lógica de negocio hacia la interfaz de usuario. Aislamiento total del dominio. Uso de Value Objects / entidades para validar las reglas de negocio aritméticas.
Interacción IA (Antigravity) No adjunta el resumen o muestra un uso pasivo (solo copió y pegó un prompt sin guiar a la herramienta). Adjunta el resumen con prompts, pero hay poca evidencia de cómo usó la IA para refactorizar iterativamente. Documenta un "Pair Programming" real con la IA. Muestra prompts claros, iterativos y correctivos enfocados en S.O.L.I.D.
